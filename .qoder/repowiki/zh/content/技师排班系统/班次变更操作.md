## 简介
本文件围绕“班次变更操作”功能进行系统化技术说明，重点解析前端 onShiftChange 方法的实现逻辑，涵盖班次类型选择、日期校验、覆盖保存机制与数据库更新流程；阐述业务规则（今日之前日期不可修改、覆盖保存、冲突检测）；说明用户交互设计（下拉选择器、实时预览、成功反馈）；解释排班记录数据结构与约束；给出错误处理策略（网络异常、权限不足、数据冲突）；并补充日志与审计建议。

## 详细组件分析

### onShiftChange 方法实现逻辑
- 输入参数
  - 事件对象包含当前行的 staffId 与列的 date
  - 事件详情包含下拉框选中的索引，映射到班次类型数组
- 日期校验
  - 若所选日期早于今日，则阻止变更并提示
- 覆盖保存
  - 先查询是否存在同 staffId 且同 date 的记录
  - 若存在则按 ID 更新；否则新增一条记录
- 界面更新与反馈
  - 成功后更新本地 scheduleMap 对应项，即时显示新标签
  - 显示成功提示；失败时隐藏加载并提示失败

```mermaid
flowchart TD
Start(["进入 onShiftChange"]) --> GetArgs["读取 staffId、date、shiftType"]
GetArgs --> CheckDate{"date < today ?"}
CheckDate --> |是| Toast["提示：不允许修改今日之前的排班"] --> End
CheckDate --> |否| Load["显示加载提示"]
Load --> Query["查询是否存在现有记录"]
Query --> Exists{"存在记录？"}
Exists --> |是| Update["按 ID 更新 shift"]
Exists --> |否| Insert["插入新记录 {date, staffId, shift}"]
Update --> Refresh["更新本地 scheduleMap"]
Insert --> Refresh
Refresh --> Success["显示成功提示"] --> End(["结束"])
Success --> End
```

### 排班记录数据结构与约束
- 数据结构
  - date: string（YYYY-MM-DD）
  - staffId: string
  - shift: ShiftType（早班/晚班/休息/请假）
- 约束
  - 同一 staffId + date 的组合在同一集合中唯一（由覆盖保存策略保证）
  - 业务上禁止修改今日之前的记录
  - UI 上通过禁用或提示限制不可操作

### 业务规则与冲突检测
- 今日之前日期禁止修改
  - 在前端进行日期比较，早于今日直接拦截
- 覆盖保存机制
  - 查询是否存在同 staffId + date 的记录
  - 存在则更新，不存在则插入
- 冲突检测
  - 当前实现基于“同一 staffId + date”的唯一性，未见显式并发冲突处理
  - 如需更强一致性，可在数据库层面建立复合索引并在云函数侧做幂等写入

### 错误处理策略
- 网络异常
  - 云数据库封装对异常进行捕获并返回空或布尔值
  - 页面层在 try/catch 中统一提示
- 权限不足

- 数据冲突

  - 可结合数据库唯一索引与云函数幂等写法进一步强化

## 故障排查指南
- 无法修改历史日期
  - 检查日期比较逻辑与 today 变量来源
  - 确认页面未对历史日期启用选择器
- 更新失败
  - 检查云数据库封装的返回值与异常捕获
  - 确认网络状态与权限
- 权限问题
  - 检查页面与按钮权限映射
  - 确认当前角色具备“管理排班”权限
