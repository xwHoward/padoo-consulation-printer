### ReservationUtils 组件分析
ReservationUtils 提供了预约相关的三大核心能力：批量删除、未来预约重分配、客户信息保存。

#### 批量删除（markReservationAsArrived）
- **输入**：预约 ID 数组
- **行为**：遍历 ID，将每个预约状态更新为"arrived"，并统计成功数量；异常时返回 0
- **事务特性**：逐条更新，无显式事务包裹；若需强一致性，建议在云函数中实现原子更新或引入事务

```mermaid
flowchart TD
Start(["进入 markReservationAsArrived"]) --> CheckIds["检查输入数组是否为空"]
CheckIds --> |为空| ReturnZero["返回 0"]
CheckIds --> |非空| Loop["遍历每个预约ID"]
Loop --> Update["按ID更新状态为 arrived"]
Update --> Count["计数+1"]
Count --> Next{"还有ID吗？"}
Next --> |是| Loop
Next --> |否| TryEnd["返回成功计数"]
TryEnd --> Catch["异常则返回 0"]
ReturnZero --> End(["结束"])
Catch --> End
```

## 性能考量
- **云函数调用**：DataLoaderService 与 ReservationUtils 均会调用 getAvailableTechnicians，建议合并请求或缓存结果以减少网络开销
- **批量更新**：markReservationAsArrived 逐条更新，大量数据时建议迁移至云函数并使用批量更新或事务
- **查询优化**：cloud-db.find 支持条件查询，避免全量扫描；对高频查询建立索引
- **冲突检测优化**：getAvailableTechnicians 使用并行查询和合并算法，提高检测效率
- **时间计算**：util 中的时间解析与比较均为常量/线性复杂度，影响较小

## 故障排查指南
- **删除失败**：markReservationAsArrived 在异常时返回 0，检查网络与权限；必要时在云函数中增加幂等与回滚
- **技师不可用**：ModalHandler 在加载可用技师失败时会提示错误，检查云函数返回结构与 code/message
- **客户信息未更新**：saveCustomerInfo 仅按电话去重，若电话不同但实为同一人会导致重复；建议增加多字段唯一性校验
- **时间冲突检测异常**：使用 isTimeOverlapping 进行冲突检测，确保 UI 提示与后端一致
- **重叠容差问题**：如果出现意外的冲突检测结果，检查 getAvailableTechnicians 中的10分钟容差设置
